# CLAUDE.md — Sable Development Guide

## Project Overview

Sable is a Matrix chat client built with React 18, TypeScript, and Vite. It runs as a web app at app.sable.moe with desktop and mobile ports in progress.

## Quick Reference

```bash
pnpm install              # Install dependencies (requires pnpm 10.32.1+, Node 24.x)
pnpm run dev              # Dev server on :8080
pnpm run build            # Production build to dist/

pnpm run lint             # ESLint
pnpm run lint:fix         # ESLint with auto-fix
pnpm run fmt:check        # Prettier check
pnpm run fmt              # Prettier format
pnpm run typecheck        # TypeScript type checking
pnpm run knip             # Detect unused exports/dependencies
pnpm test                 # Vitest in watch mode
pnpm test:run             # Vitest single run (CI)
```

**All of these must pass before any PR:** `fmt:check`, `lint`, `typecheck`, `knip`, `test:run`

## Tech Stack

- **UI:** React 18 + TypeScript 5.9 + Vite 7
- **Styling:** Vanilla Extract (zero-runtime CSS-in-JS)
- **State:** Jotai (atomic state) + TanStack React Query (server state)
- **Matrix:** matrix-js-sdk 38.x
- **Editor:** Slate (rich text)
- **i18n:** i18next (translations in `public/locales/`)
- **Testing:** Vitest + React Testing Library

## Code Style

- Prettier: 100 char width, single quotes, ES5 trailing commas
- ESLint: airbnb-extended config with React/TypeScript rules
- Do not add comments, docstrings, or type annotations to code you didn't change

## Cross-Platform Architecture

Sable is being ported to **Windows** (via Tauri) and **Android** (via Capacitor). The core principle is: **platform-specific code lives in its own directory and never leaks into `src/`.**

### Directory Layout

```
SableD/
├── src/                   # Shared web core — DO NOT put platform code here
├── platforms/
│   ├── tauri/             # Windows desktop app (Tauri v2)
│   │   ├── src-tauri/     # Rust backend (Cargo.toml, main.rs, commands, etc.)
│   │   ├── tauri.conf.json
│   │   └── README.md
│   └── capacitor/         # Android app (Capacitor)
│       ├── android/       # Native Android project (generated by Capacitor)
│       ├── capacitor.config.ts
│       └── README.md
├── src/platform/          # Thin platform abstraction layer (see below)
│   ├── index.ts           # Re-exports the active platform adapter
│   ├── types.ts           # PlatformAdapter interface
│   ├── web.ts             # Web/browser adapter (default)
│   ├── tauri.ts           # Tauri adapter (uses @tauri-apps/api)
│   └── capacitor.ts       # Capacitor adapter (uses @capacitor/ plugins)
```

### Rules for Platform Code

1. **`src/` is platform-agnostic.** No direct imports of `@tauri-apps/*` or `@capacitor/*` in `src/app/`, `src/client/`, or any shared module. If you need a platform capability, use the adapter in `src/platform/`.

2. **`src/platform/` is the only bridge.** This thin abstraction layer exposes a `PlatformAdapter` interface for capabilities that differ across platforms (notifications, file system, deep links, window management, etc.). Each platform implements this interface. The active adapter is selected at build time via Vite environment variables or dynamic detection.

3. **`platforms/tauri/` owns all Tauri config and Rust code.** Tauri's Rust backend, `tauri.conf.json`, and any Tauri-specific build scripts live here. The Tauri app loads the same Vite dev server or production build as the web app.

4. **`platforms/capacitor/` owns all Capacitor config and native Android code.** The generated `android/` project, `capacitor.config.ts`, and any native plugins live here. Capacitor wraps the web build into a native Android WebView.

5. **No `#ifdef`-style platform branching in shared code.** If a feature only exists on one platform, gate it at the feature level using the platform adapter's capability checks, not by scattering conditionals through components.

6. **Platform-specific UI components** (e.g., a Windows title bar, Android back button handler) go in `src/app/components/platform/` and are lazy-loaded only on the relevant platform.

### Platform Adapter Pattern

```typescript
// src/platform/types.ts
export interface PlatformAdapter {
  readonly name: 'web' | 'tauri' | 'capacitor';
  showNotification(title: string, body: string): Promise<void>;
  // Add capabilities as needed: file access, deep links, window controls, etc.
}
```

```typescript
// src/platform/index.ts — resolved at build time or with runtime detection
export { adapter as platform } from './web';  // default; overridden per platform build
```

Shared code imports only from `src/platform`:
```typescript
import { platform } from '../platform';
await platform.showNotification('New message', body);
```

### Build Targets

| Platform  | Build Command              | Output                    |
|-----------|----------------------------|---------------------------|
| Web       | `pnpm run build`           | `dist/` (static files)    |
| Windows   | `cd platforms/tauri && pnpm tauri build` | `.msi` / `.exe` installer |
| Android   | `cd platforms/capacitor && npx cap sync && npx cap open android` | Android Studio project |

Each platform build consumes the same Vite-built web assets. Tauri and Capacitor are wrappers, not forks.

### What NOT to Do

- Do not import `@tauri-apps/api` or `@tauri-apps/plugin-*` anywhere in `src/` outside of `src/platform/tauri.ts`
- Do not import `@capacitor/*` anywhere in `src/` outside of `src/platform/capacitor.ts`
- Do not add platform-specific dependencies to the root `package.json` — use workspace packages or conditional dependencies scoped to the platform directory
- Do not modify `index.html`, `vite.config.ts`, or shared build config for a single platform — use platform-specific Vite config overrides in the platform directory if needed
- Do not put native Android (Java/Kotlin) or Rust code in `src/`
